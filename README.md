# HF Reviewer Finder

Find and visualize the most topically relevant reviewers for a *Human Factors*
journal manuscript. Paste a title + abstract; the app maps the journal's reviewer
pool in a 2D semantic space, shows where your manuscript lands, and ranks
reviewers by similarity to it.

![concept](https://img.shields.io/badge/stack-Streamlit%20%2B%20Altair%20%2B%20sentence--transformers-blue)

## How it works

1. Each reviewer in `data/members_enriched.csv` is turned into a text **profile**
   (expertise overview + their 3 recent and 2 seminal publications with synopses).
2. Profiles are embedded locally with `all-MiniLM-L6-v2` (cached to `data/.cache/`).
3. Your manuscript text is embedded the same way; **cosine similarity** ranks the
   reviewers, and **PCA/UMAP** projects everyone (including your text) into a 2D map.

The matching engine lives in `engine.py` (pure Python, no Streamlit) so it can be
scripted or tested independently.

## Setup

```bash
cd ~/_Code/HF-Reviewer-Finder
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

First run downloads the embedding model (~90 MB) once.

## Pre-calculate artifacts (fast startup)

Run once so the app loads instantly — this precomputes and caches to disk the
embeddings, **both the PCA and UMAP co-fit projections** (the UMAP fit is the
slow step, ~11 s), and the default-manuscript embedding:

```bash
python precompute.py
```

After this the app starts in ~1 s with no model load and no projection fitting;
it just reads the cached artifacts from `data/.cache/`. The current artifacts are
**committed to the repo** (~2 MB) so a fresh clone/deploy is fast out of the box.
Caches are keyed by embedding **content**, so re-running `precompute.py` after you
change `members_enriched.csv` or add abstract batches rebuilds only what changed;
then commit the regenerated `data/.cache/*` artifacts.

To rebuild the **UMAP** coordinates you need `umap-learn` installed locally (it's
not in `requirements.txt`): `pip install "umap-learn>=0.5,<0.6"`, then
`python precompute.py`. Without it, `precompute.py` falls back to PCA coords for
the UMAP map.

## Run

```bash
streamlit run app.py
```

Then open the local URL Streamlit prints (usually http://localhost:8501).

## CLI smoke test (no browser)

```bash
python engine.py "drivers' trust in automated vehicle takeover requests"
```

Prints the top-10 reviewer matches — useful to confirm the engine works before
launching the UI.

## Data

`data/members_enriched.csv` — 163 *Human Factors* reviewers with role code, expertise
overview, and key publications. 75 come from the editorial board (codes AE/EB/R/PR),
generated by the `HF_Review` Claude skill (Power BI scrape + web research). The other
88 carry the code **`Not_EB`**: prolific authors (>3 papers in the
`Human_Factors_*abstracts*.json` corpus) who are *not* on the editorial board, added
as candidate reviewers. **Citations are LLM-researched — verify before formal/editorial
use.** Regenerate the editorial-board rows by re-running `HF_Review`.

The `Not_EB` rows are (re)built by two helper scripts:
- `enrich_metadata.py` first adds `citation_count` + `authors_orcid` to the abstract
  batches (see below).
- `python aggregate_not_eb.py` rebuilds every `Not_EB` row from the per-author profiles
  in `data/not_eb_parts/*.json` (each profile produced by web research with live DOI
  verification; `data/not_eb_parts/AGENT_SPEC.md` is the research spec). It is
  idempotent — editorial-board rows are never touched — so it is safe to re-run as
  profiles are added or corrected. Run `python precompute.py` afterward to refresh the
  embeddings/projection cache so the app picks up the new reviewers.

`data/Human_Factors_*abstracts*.json` — recent *Human Factors* article abstracts
(currently **1,000** papers across two batches) shown as the grey background field.
Drop in more `Human_Factors_*abstracts*.json` batches and they're picked up
automatically (re-run `precompute.py` afterward).

Each record carries: `title`, `authors` (comma-separated string), `date`, `doi`,
`abstract`, plus three Crossref-enriched fields added by `enrich_metadata.py`:
`citation_count` (int, Crossref `is-referenced-by-count`), `authors_orcid` (list of
`{name, orcid}` per author, `orcid` null when unavailable), and `enriched_date`
(ISO date the citation count was captured — counts drift, so re-run periodically).
Run `python enrich_metadata.py` after adding new batches; it's resumable and only
fetches records missing `citation_count` (use `--force` to refresh all). ORCID
coverage is partial (~66% of papers have ≥1 author ORCID; ~28% of all authors);
citation counts cover 100% of papers (0 for the ~11% not yet cited).

## Deployment

Tested on Streamlit Community Cloud (repo: `app.py`, branch `master`).

- **Numba-free at runtime.** The UMAP map is served from precomputed coordinates
  (committed `.npz`) and the manuscript marker is placed with a numpy k-NN method,
  so the deployed app never imports `umap`/`numba`. `umap-learn` is therefore *not*
  a runtime dependency (it's only needed locally to rebuild UMAP coords). This
  avoids numba's lag behind new Python releases — e.g. **Python 3.14**, where numba
  cannot JIT and a UMAP `.transform` would crash.
- **Python version:** for extra safety, set Python **3.13** in the Streamlit Cloud
  app's *Advanced settings* (Cloud does not read `.python-version`). The app also
  runs on 3.14 thanks to the numba-free design.
- **Pinned dependencies** (`requirements.txt`) keep deploys reproducible. The stack
  includes `torch` (via sentence-transformers), so the install is large
  (~hundreds of MB) and needs a host with adequate RAM.
- **Committed artifacts** mean a cold start does *not* download the model, embed
  1,075 texts, or fit any projection at first request — it reads `data/.cache/*`.
  Only a *custom* manuscript (different from the default) lazily loads the model to
  embed it, so a deploy that should never load the model can keep to the default.
- The cache directory is written lazily and **degrades gracefully on a read-only or
  ephemeral filesystem** (recomputes in memory instead of crashing).
- Missing/malformed data files surface a clear in-app error rather than a stack trace.

## Notes

- UMAP is optional at build time; if `umap-learn` isn't installed when (re)building
  coords, the UMAP map falls back to PCA coordinates automatically.
- Embeddings and projections are cached by a content hash, so editing the CSV (even
  with the same number of rows) triggers a one-time re-embed and re-projection.
